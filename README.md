# HandoffBox

> Yesterday's self hands today's self the next step.

HandoffBox is not another productivity app.

It is a simple operating principle for reducing the "restart tax" of unfinished work.

## The Problem

As solo builders, we often stop at 80%.

Not because of a lack of skill or time, but because every restart requires reconstructing context:

- Where did I stop?
- Why did I stop?
- What should I do next?

The cost of remembering becomes the reason projects never finish.

## The Solution

Before stopping work, leave a handoff:

```md
## Stopped at

## Why I stopped

## Next step (within 15 minutes)
```

Tomorrow, simply execute the next step.

When finished, close the loop.

## Why GitHub Issues?

HandoffBox was originally imagined as a dedicated application.

Then we realized GitHub Issues already had everything we needed:

- Open
- Comment
- Pin
- Notifications
- Close

The missing piece wasn't another tool.

It was the habit of leaving a handoff.

## Philosophy

Build less.
Finish more.

Reduce restart tax.
Close loops.

Yesterday's self should help today's self.

## Workflow

Issue → Work → Leave handoff → Resume → Close

That's it.

# HandoffBox

> 昨日の自分から今日の自分へ、次のステップを引き継ぎます。

HandoffBoxは、単なる生産性向上アプリではありません。

未完了の作業による「再開コスト」を削減するためのシンプルな運用原則です。

## 問題点

一人で作業する人は、しばしば80%のところで作業を中断してしまいます。

スキルや時間の不足が原因ではなく、再開するたびに状況を再構築する必要があるからです。

- どこで作業を中断したのか？

- なぜ中断したのか？

- 次に何をすべきか？

これらのことを思い出すコストが、プロジェクトがいつまでも終わらない原因となります。

## 解決策

作業を中断する前に、引き継ぎを残しましょう。

```md
## 中断した箇所

## 中断した理由

## 次のステップ（15分以内）
```

明日は、次のステップを実行するだけです。

完了したら、ループを閉じます。

## なぜGitHub Issuesを使うのか？


HandoffBoxは当初、専用アプリケーションとして構想されました。

しかし、GitHub Issuesには既に必要な機能がすべて備わっていることに気づきました。

- オープン
- コメント
- ピン留め
- 通知
- クローズ

欠けていたのは、別のツールではありませんでした。

それは、ハンドオフを中断する習慣でした。

## 理念

構築は少なく。

完了を多く。

再始動の負担を軽減。

ループを閉じる。

昨日の自分が今日の自分を助ける。

## ワークフロー

課題 → 作業 → ハンドオフを中断 → 再開 → クローズ

以上です。
